69. Webpack 和 Vite 的区别
1. 技术栈与底层原理
- Webpack
  - 使用 JavaScript 写成，运行在 Node.js 上
  - 文件读写和打包速度相对较慢
  - 基于 打包（bundle） 的思想：从入口文件 entry 出发，递归构建 依赖图（dependency graph），通过loader或plugin处理，然后打包成一个或多个 bundle 输出
- Vite
  - 使用 esbuild（Go 语言） 进行开发时的预构建和转译，比 Webpack 快 10~100 倍
  - 基于 ES Module（ESM）+ 按需加载
    - 开发阶段几乎不打包，浏览器按需请求模块，Vite 实时编译对应文件
    - 对第三方依赖（如 Vue、ECharts）提前预构建成浏览器可直接识别的 ESM（存放在 node_modules/.vite），避免重复编译
  - 模块机制差异：
    - Webpack：必须事先构建依赖图，把所有模块打包到 bundle
    - Vite：浏览器直接按需请求模块，开发模式无需完整打包
2. 热更新（HMR）
  - Webpack：修改模块时，需要重新打包该模块及其依赖，再注入浏览器，项目大时会卡
  - Vite：只编译修改的单个文件，更新速度几乎与项目大小无关
3. 配置复杂度
  - Webpack：
    - 配置灵活但复杂，可以处理各种资源
    - 需要 Loader（如 babel-loader、css-loader）和 Plugin（如 HtmlWebpackPlugin、MiniCssExtractPlugin）
    - 插件生态丰富，适合复杂项目，大型项目 webpack.config.js 常有数百行
  - Vite：
    - 内置常用功能（JSX、CSS、图片资源等），基础项目零配置即可运行
    - 配置简单，vite.config.js 通常不超过 50 行
4. 构建工具与生产打包
  - Webpack：
    - 打包器和构建系统一体
    - 构建大项目时较慢
    - 对 ES6+ 语法需配合 @babel/core、core-js 兼容旧浏览器
    - 优化需手动配置各种 Plugin
  - Vite：
    - 开发阶段 → 使用 esbuild，转译极快
    - 生产阶段 → 使用 Rollup 打包，优化能力强，支持 Tree Shaking
    - 打包 ES Module 更高效，输出 bundle 更小，默认优化现代浏览器兼容性
5. 适用场景
  - Vite：
    - 基于 Vue3、React18 等现代框架的项目
    - 中小型项目，追求开发效率
  - Webpack：
    - 需支持 IE11 及以下老旧浏览器的项目
    - 大型项目或有复杂构建需求（自定义 Loader/Plugin、特殊工具链）
    - 基于传统框架（如 Vue2、AngularJS）项目

1. 项目脚手架与结构化：项目初始化（脚手架）、目录与模块设计、代码规范

71. Tree Shaking 是什么？
1. 定义：在打包时自动删除项目中未使用的代码，只保留真正被用到的部分。依赖 ES Module（ESM）静态结构
2. 使用前提：使用import/export
3. 在构建配置中启用 Tree Shaking（Vite 默认开启）
72. 什么时候使用请求合并和延迟加载？
1. 请求合并：将多个请求合并成一个请求，减少 HTTP 开销 —— 减少网络开销
2. 延迟加载：推迟资源或数据的加载时间，等到真正需要时再加载 —— 增加首屏加载速度
3. 使用时机判断
  请求合并：
    1. 多个请求数据结构相似 —— 减少网络开销
    2. 多个请求依赖同一数据 —— 减少重复开销
  延迟加载
    - 不确定用户是否会点击的内容 —— 避免无效请求
    - 用户滚动到底部才看到的图片/模块 —— 提升首屏性能
    - 动态路由或次级页面资源 —— 降低初始包体积
73. 什么是骨架屏
骨架屏是指在页面内容尚未加载完成时，展示一个简化的“灰色占位结构”，让用户感觉加载更流畅。
使用时机：
  - 页面或模块数据加载较慢时（如首屏、详情页）。
  - 图片或列表大量加载时。
  - 内容异步渲染时（SSR + 客户端激活）。

75. 懒加载
1. 定义：延迟加载资源或组件，直到真正需要时才加载（核心思想：按需加载 + 延迟执行）
2. 优点：
  - 减少首屏体积（只加载必要内容）
  - 减少网络请求峰值（按需加载资源）
  - 提升用户体验（页面更快可见、更流畅）
3. 使用时机
  - 图片或视频较多的长页面；
  - 大型 SPA（单页应用）；
  - 路由切换频繁的项目；
  - 某些资源“可能不会被访问”时。
4. 常见使用情况
  1. 图片懒加载：图片初始不加载真实资源，只有当图片即将出现在视口内时，再替换真实 src 去加载。
    1. 实现方式：IntersectionObserver、原生属性（loading= "lazy"）
// 原理：IntersectionObserver 检测元素是否进入视口，在进入时触发 src 替换，实现懒加载。
const imgs = document.querySelectorAll('img.lazy')
const observer = new IntersectionObserver((entries, observer) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) { // 元素进入可视区
      const img = entry.target
      img.src = img.dataset.src // 加载真实图片
      observer.unobserve(img)   // 停止监听
    }
  })
})

imgs.forEach(img => observer.observe(img))
  2. 组件懒加载：不是在首屏时就加载，而是在用户真正访问时通过动态 import() 异步加载。
const UserPage = () => import('@/views/UserPage.vue') // Vue实现
// React 实现

import React, { lazy, Suspense } from 'react'
const UserPage = lazy(() => import('./UserPage'))
  3. 路由懒加载：把页面组件的加载和路由访问绑定起来，只有当用户访问该路由时才加载对应页面。
// Vue Router 实现
import { createRouter, createWebHistory } from 'vue-router'

const routes = [
  {
    path: '/home',
    component: () => import('@/views/Home.vue')  // 懒加载
  },
  {
    path: '/about',
    component: () => import('@/views/About.vue') // 懒加载
  }
]

export default createRouter({
  history: createWebHistory(),
  routes
})

// React Router 实现
const Home = lazy(() => import('./pages/Home'))
const About = lazy(() => import('./pages/About'))

<Routes>
  <Route path="/home" element={<Home />} />
  <Route path="/about" element={<About />} />
</Routes>

77. vite插件的生命周期
┌───────────────────────┐
│ config()              │ ← 读取 vite.config.ts 前
└──────────────┬────────┘
               │
┌──────────────▼────────┐
│ configResolved()      │ ← 配置解析完成后
└──────────────┬────────┘
               │
┌──────────────▼────────┐
│ buildStart()          │ ← 打包启动
│  ↓                    │
│ transform(code, id)   │ ← 转换每个模块
│  ↓                    │
│ generateBundle()      │ ← 打包阶段产物生成
│  ↓                    │
│ writeBundle()         │ ← 输出阶段，写文件
└───────────────────────┘

不同阶段可以实现不同功能：
- config() / configResolved()：读取配置、修改 Vite 设置；
- buildStart()：扫描文件或初始化；
- transform()：修改源码；
- generateBundle()：分析打包结果；
- writeBundle()：生成报告、上传、通知等。

78. 实现性能优化的思路
1. 性能优化的三大核心目标：
  1. 加载更快 ——> 减少首屏加载速度，减少请求次数，减少资源体积
  2. 渲染更快 ——> 页面流畅运行，减少重排重绘、合理使用虚拟列表、优化动画
  3. 交互流畅 ——> 避免卡顿，优化事件响应、减少内存占用
2. 整体思路: 先量化 ——> 找瓶颈 ——> 局部优化 ——> 验证效果 ——> 自动化监控
  1. 发现问题（监测）：通过测试时用户体验 + 工具监测（performance、Lighthouse、PageSpeed Insights）输出一个报告或者性能指标，找到性能瓶颈
  2. 分析原因（诊断）：通过DevTools、bundle-analyzer、memory等工具输出问题清单，找根本问题
  3. 制定方案（优化）：通过缓存、懒加载、压缩、分包等来输出优化计划，选择最好的策略来实现性能优化
  4. 验证效果（评估）：再次使用性能工具进行测试，对比优化前后的数据来验证优化效果
3. 具体实现
  1. 加载阶段
    思路：减少资源体积 + 提高网络利用率
    - 压缩 JS/CSS/HTML。
    - 按需加载（懒加载 / 代码分割）。
    - 图片压缩、用 WebP。
    - 利用 CDN、HTTP 缓存。
    - SSR/SSG 提前输出首屏。
    🧩 关键指标：
    - FCP（First Contentful Paint）首次内容绘制（<1.8）：
      - 定义：浏览器渲染出第一个内容元素的时间
      - 关注点：页面开始加载的初始响应速度
    - LCP（Largest Contentful Paint）最大内容绘制
      - 定义：首屏中最大内容元素渲染完成的时间
      - 关注点：用户感知到的页面加载速度和可用性
      - 良好 < 2.5 秒，> 4s 差
  2. 渲染阶段
    思路：减少浏览器工作量
    - 优化 DOM 结构（层级浅、节点少）。
    - 合理使用虚拟 DOM 或 diff。
    - 批量修改 DOM，避免频繁回流。
    - 用 CSS3 动画替代 JS 动画。
    - 虚拟列表加载大量元素。
    🧩 关键指标：
    - FPS（每秒帧数）
    - Layout / Paint 时间占比
  3. 交互阶段
    思路：保证响应迅速、不卡顿
    - 避免主线程被长任务阻塞。
    - 使用 Web Worker 处理耗时任务。
    - 减少不必要的事件监听。
    - 使用节流、防抖。
    - 优化滚动与输入体验。
    🧩 关键指标：
    - TBT（Total Blocking Time）阻塞总时间
    - INP（Interaction to Next Paint）

79. 加载性能优化
让网页更快打开、首屏更快可见。
1. 资源体积优化
  - 代码压缩与混淆
    - 使用 Vite/Webpack 的 terser 或 esbuild 压缩。
  - Tree Shaking
    - 删除未使用的模块（ESM 必须是静态导入）。
  - 代码分割 (Code Splitting)
    - 按路由、组件或功能拆分 bundle，懒加载。
    - const Page = () => import('@/views/Page.vue')
  - 图片优化
    - 用 WebP、AVIF 替代 PNG/JPG。
    - 使用 srcset 响应式加载。
    - 雪碧图 / iconfont / SVG Sprite。
  - 字体优化
    - 精简字体子集，使用 font-display: swap;。
2. 网络请求优化
  - HTTP 缓存（强缓存 + 协商缓存）
    - Cache-Control, ETag, Last-Modified。
  - CDN 加速
    - 静态资源走 CDN，减轻服务器负担。
  - 请求合并与延迟加载
    - Debounce、节流。
    - GraphQL / REST 接口聚合。
3. 首屏加载优化
  - SSR / SSG（如 Nuxt、Next）
    - 提前在服务器渲染 HTML。
  - 骨架屏 + Loading
    - 提升用户感知速度。
  - 预加载与预获取
    - <link rel="preload" href="/main.js" as="script">
<link rel="prefetch" href="/next-page.js">
80. 渲染性能优化
让页面流畅运行、不卡顿。
1. 减少重排与重绘
  - 批量修改 DOM（使用 DocumentFragment 或虚拟 DOM）。
  - 避免频繁读写 offset、scroll 等属性。
  - 用 CSS3 动画代替 JS 动画（GPU 加速）。
  - 合理使用 will-change。
2. 合理使用虚拟列表（大数据渲染）
  - 用虚拟滚动（如 VueVirtualScroller、React Window）。
  - 表格、长列表等只渲染可见区域。
3. 避免性能陷阱
  - 不要在模板中使用复杂计算。
  - 使用 computed/memo 缓存结果。
  - 减少 watch 深层监听。
81. 代码层面优化
1. 合理使用异步
  - 利用 requestIdleCallback / requestAnimationFrame。
  - 大任务切分为小任务，防止主线程阻塞。
2. 减少内存泄漏
  - 清理定时器、事件监听、闭包变量。
  - Vue/React 组件销毁时释放资源。
3. 使用现代特性
  - HTTP/2 / HTTP/3 并行传输。
  - ES Modules 原生支持（浏览器端）。
