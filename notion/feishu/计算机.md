42. HTTP的请求方式
1. Get：用于获取资源，通过URL传递参数，请求结果会被缓存，可以被书签保存，不适合传递敏感信息
2. post：用于提交资源，将数据放在请求体中发送给服务器，请求的结果不会被缓存
3. put：用于更新资源（通常用于更新整个资源），将数据放在请求体中发送给服务器
4. delete：用于删除指定资源，将数据放在请求体中发送给服务器
5. patch：用于部分更新资源（通常用于更新资源的部分属性），将数据放在请求体中发送给服务器
43. get和post的区别
1. 区别：
  1. get是幂等（多次访问结果一样），post不是
  2. get能触发浏览器缓存，post不会
  3. get能由浏览器自动发起（如img-src，资源加载），post不行
  4. post相对安全
2. 相同点：
  1. 都不安全，都基于Http，明文传输
  2. 参数并没有大小限制，是URL大小有限制，因为要保护服务器
44. RESTful规范
1. 是一种用于Web API的设计风格
  1. 核心思想：资源（Resource）为核心，通过 统一的接口（HTTP 方法） 来操作资源。
  2. 用语义化的URL来表示资源的层级关系和操作，如/user表示用户资源
  3. 无状态：每个请求都是独立的，服务器不保存客户端的状态信息
2. 规范：
  1. 使用 资源路径（URI） 表示对象
  2. 使用 HTTP 方法 表示操作
  3. 使用 状态码 表示结果
  4. 一般用JSON作为数据格式
45. 常见的HTTP状态码和代表的意义
- 200：请求成功，服务器成功处理了请求
- 201：请求成功，并在服务器哈桑创建了新的资源
- 204：服务器成功处理了请求，但是没有返回任何内容
- 400：服务器无法理解请求的语法，请求有语法错误（参数不正确或者格式错误）
- 401：请求需要用户身份验证（客户端未提供或者提供了无效的身份验证令牌、用户名和密码等）
- 403：服务器拒绝请求，没有访问权限（用户没有访问权限来请求资源，或者请求的资源被设置为禁止访问）
- 404：请求的资源不存在
- 405：请求方法不被允许
- 500：服务器内部错误，无法完成请求
- 502：服务器作为网关或代理，从上游服务器收到无效响应
- 503：服务器当前无法处理请求，通常由于过载或维护
46. 301、302、303的区别？
- 301 永久重定向。在下次访问旧的URL时，浏览器会直接跳转到新的URL，不会再向服务器请求
- 302 临时重定向。每次访问旧的URL，都会向服务器请求，然后服务器返回新的URL
- 303 查看其他位置。表示请求的资源存在于另一个URL，应使用GET方法获取。这个状态码主要用于在执行POST、PUT等可能引起服务器状态变化的操作后，将客户端重定向到一个新的资源，避免用户刷新或重复提交表单
47. HTTP和HTTPS的区别
主要区别在安全性和数据传输方式上
1. 安全性：
  1. HTTP协议传输的数据都是未加密的，明文数据，可以被任何抓包工具截取并查看
  2. HTTPS协议由SSL+HTTP构建的，可以进行加密传输、身份认证的网络协议，更安全
2. 数据传输方式：HTTP端口号80，HTTPS端口443
3. 网址导航栏显示：http:// 和 https://
4. 证书：HTTPS需要到CA申请证书，需要一定费用
5. 网络速度：HTTP比HTTPS快，因为HTTPS要进行加密和解密
6. SEO优化：搜索引擎会更倾向于把HTTPS放在前面
48. 什么是代理服务？正向代理和反向代理
代理服务
1. 定义：是一个中间服务器，代替客户端和服务器取请求另一个服务器
2. 作用：
  - 转发请求和响应；
  - 过滤或缓存内容；
  - 改写请求头、隐藏真实 IP；
  - 做负载均衡、安全控制等。
正向代理
1. 定义：客户端找代理去访问服务器
2. 原理图：[客户端] → [正向代理服务器] → [目标服务器]
3. 特点：客户端知道目标服务器，服务器不知道客户端
4. 使用场景：
  1. 科学上网
  2. 隐藏客户端真实IP（比如公司员工上网，统一通过代理服务器转发，外网看到的都是代理服务器的 IP。）
  3. 缓存内容，加速访问（代理可缓存静态资源，下次访问更快。）
反向代理
1. 定义：服务器端用代理来接收客户端请求。
2. 原理图：[客户端] → [反向代理服务器] → [内部真实服务器]
3. 特点：客户端不知道后端服务器是谁
4. 使用场景：
  1. 负载均衡（Nginx 接收所有请求，再分发给多台后端服务器（A/B/C）。）
  2. 隐藏真实服务器（防止外部直接攻击内部服务器。）
  3. 统一接入层（代理可做 HTTPS 终止、压缩、缓存、限流等。）
49. 浏览器输入URL到页面加载显示完成全过程
用户输入 URL → 浏览器解析 URL → DNS 解析 → 建立 TCP 连接（三次握手）
→ 发送 HTTP 请求 → 服务器处理并返回响应
→ 浏览器解析 HTML → 构建 DOM / CSSOM → 渲染页面
→ 执行 JS → 加载资源（图片/字体等）→ 页面显示完成
1. 用户输入 URL
2. 浏览器解析 URL：https://www.example.com:443/path/index.html?a=1#section
  1. 浏览器会解析出以下内容：
    - 协议：https
    - 域名：www.example.com
    - 端口：443
    - 路径：/path/index.html
    - 查询参数：a=1
    - 哈希：#section
  2. 浏览器查缓存
[图片]
3. DNS解析
  1. 浏览器查找本地DNS缓存，以查找域名对应的IP地址。
  2. 未找到，浏览器将查询操作传递给操作系统的DNS解析器,操作系统将查询操作传递给本地域名服务器。
  3. 本地域名服务器在自己的DNS缓存中查找域名对应的IP地址,如果找到则返回,否则继续查询更高级别的DNS服务器,直到找到IP地址。
  4. 返回给服务器并缓存起来（通常几分钟到几小时）
4. 建立TCP链接，确保双方的发送和接收能力正常
TCP三次握手
客户端：发送 SYN=1 的包（我想建立连接）进入SYN_SENT（状态：同步已发送）
服务端：返回 SYN + ACK 包（可以建立）进入 SYN_RECEIVED 状态
客户端：回复 ACK（确认）客户端发出包进入ESTABLISHED（连接已建立） 状态，服务器接收之后也进入该状态
→ 建立连接成功

客户端（Client）                     服务端（Server）
      |                                     |
(1)   | ---- SYN=1, seq=x ------------->    |  ← 第一次握手
      |                                     |
(2)   | <---- SYN=1, ACK=1, seq=y, ack=x+1  |  ← 第二次握手
      |                                     |
(3)   | ---- ACK=1, seq=x+1, ack=y+1 ---->  |  ← 第三次握手
      |                                     |
      |        【连接建立成功】                |

如果是HTTPS，还要在TCP上建立TLS握手（多1～2个往返时间）：
  - 验证服务器证书；
  - 协商加密算法；
  - 生成会话密钥。
  1. 为什么需要三次握手？
    1. 一次——客户端能发
    2. 2次——双方能发，不确定客户端的接收能力
    3. 3次——双方都能发送接收
5. 发送HTTP请求
6. 服务器处理请求
  服务器（如 Nginx、Node.js、Java、Go）：
  1.  接收请求；
  2. 解析 URL；
  3. 查询数据库或调用业务逻辑；
  4. 生成响应（HTML / JSON / 图片等）；
  5. 通过 TCP 返回给浏览器。
7. 浏览器接收响应
浏览器渲染阶段
8. 接收HTML,构建DOM树:
  - 浏览器接收HTTP响应,并开始解析响应数据,查找HTML内容。
  - 解析HTML内容,构建DOM(文档对象模型)树,表示页面的结构。
9. 构建CSSOM树:
  - 同时,浏览器开始解析CSS样式表,构建CSSOM(CSS对象模型)树,表示页面的样式信息。
10. 渲染树构建:
  - 浏览器将DOM树和CSSOM树结合起来,构建渲染树(RendeerTree),表示页面的可见内容。
11. 布局和绘制:
  - 浏览器计算渲染树中每个元素的大小和位置,然后进行页面布局(layout)
  - 最后,浏览器绘制页面上的每个元素,呈现给用户。
12. 交互和JavaScript:
  - 如果遇到了JavaScript代码,浏览器会执行JavaScript,可以修改DOM和CSSOM,以及处理用户交互。
13. 完成页面加载:
  - 当资源(包括嵌入的资源,样式表、脚本等)加载完成,浏览器会触发 onload事件,表示页面加载完成。
14. 页面渲染完成:
  - 用户可以看到页面内容,页面渲染完成。
