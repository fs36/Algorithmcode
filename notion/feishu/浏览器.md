50. 重绘、回流（重排）？如何减少发生次数
1. 重绘：元素样式发生变化（元素的背景、字体颜色等），在不影响布局的情况下，浏览器重新绘制元素的过程
2. 回流（重排）：当页面的结构或几何属性（大小、位置）发生变化时，浏览器必须重新计算元素的布局，然后再重新渲染。
3. 回流的成本比重绘高，回流一定会触发重绘，而重绘不一定会触发回流
4. 如何减少回流？
  1. 合并多次的DOM操作：不频繁修改样式，用class一次性修改
  2. 批量修改 DOM：使用 DocumentFragment（文档片段） 批量插入 DOM 节点。在需要 批量插入/修改 DOM 的时候，创建文档片段const fragment = document.createDocumentFragment();，将这些元素添加到文档片段里面fragment.appendChild(li); ，再将这个文档片段一次性插入到DOM中DOM元素.appendChild(fragment);，减少回流次数。
  3. 使用 requestAnimationFrame ：浏览器会在下一帧渲染时统一执行，可以确保动画在浏览器的重绘周期内执行，从而避免不必要的回流
    1. requestAnimationFrame作用：告诉浏览器 在下一次重绘之前执行一个指定的回调函数，用来更新动画效果
  4. 使用visibility：hidden代替display：none：visibility：hidden不会触发回流，因为元素仍占据空间，只是不可见，display：none会将元素从渲染树移除，引起回流
  5. 让元素脱离文档流：修改元素前先 display: none，改完再显示 或者用 position: absolute/fixed 脱离文档流
  6. 减少强制同步布局：避免频繁读取 offsetHeight/getComputedStyle 后立刻修改 DOM，如果必须读取，尽量先缓存
5. 如何减少重绘？
  1. 用CSS动画代替JS动画：用transform和opacity来创建动画效果，不是改变元素的布局属性（宽、高等）。例如改变位置用transform：tanslate(x,y)，不用top/left，改变大小用scale(对元素进行缩放)，而不是width/height
51. 浏览器的渲染流程
1. 处理HTML并构建DOM树
  1. 浏览器开始解析之前，开启一个与解析线程，率先下载HTML外部的CSS、JS文件
  2. 遇到JS代码，暂停解析HTML，等待JS文件下载完并全局代码解析完成之后才能继续解析HTML
  3. 可以修改 DOM 或 CSSOM，所以执行 JS 可能会触发回流（reflow/repaint）。
  4. 优化方式：
    - <script defer>：推迟到 HTML 解析完成后执行，不阻塞 DOM 构建。
    - <script async>：异步加载，加载完成后立即执行，可能打断 DOM 解析。

    ↓ 遇到 JS

    ↓ 
DOM + CSSOM → 渲染树 → 布局（Layout） → 绘制（Paint）
1. CSS 的阻塞
  - 下载 CSS 文件时不会阻塞 HTML 的解析（DOM 继续构建）。
  - 但是！CSS 的下载和解析会阻塞 JS 的执行。
  - 👉 因为 JS 可能依赖 CSSOM（例如 window.getComputedStyle()）。
2. JavaScript 的阻塞
  - 默认 <script> 会阻塞 DOM 的解析和渲染，直到脚本下载并执行完毕。
  - 解决方案： 
    - defer：脚本下载与 HTML 解析并行，等 DOM 构建完成后再顺序执行。
    - async：脚本下载与 HTML 解析并行，下载完成后立即执行（可能打乱顺序）。
52. 浏览器缓存（强缓存/协商缓存）
若缓存生效，强缓存返回200，协商缓存返回304
浏览器缓存的目的：减少重复请求，提高页面加载速度。
1 强缓存（效率高，静态资源）
1. 特点：在缓存有效期内，浏览器直接用本地缓存，不发请求到服务器。
2. 实现方式：在HTTP响应头中有 Expires 或 Cache-Control，根据 Expires 或 Cache-Control 判断是否命中强缓存
  1. Expires（HTTP/1.0 过时）：Expires:new Date(Date.now()+10*1000).toUTCString表示缓存到期时间
  2. Cache-Control (推荐 HTTP/1.1)：Cache-COntrol: max-age=3600，资源在3600秒内都是可以直接获取，不需要请求服务器
2 协商缓存（接口数据）
1. 特点：浏览器先发请求问服务器，服务器决定是否用缓存。如果资源没变，返回 304 Not Modified，浏览器用本地缓存；如果变了，返回新资源。
2. 实现方式：HTTP响应头和请求头配合
  1. Last-Modified / If-Modified-Since（检测资源的最后修改时间）
    1. 响应头：Last-Modified：Wed， 30 Sep 2025 10:00:00 GMT
    2. 请求头：If-Mpdified-Since：wed，30 Sep 2025 10:00:00 GMT
    3. 服务器对比浏览器发过来的If-Mpdified-Since 和自己的Last-Modified，如果没改返回304
    4. 缺点：1. 时间精度秒级，不够细致；2. 文件频繁修改但内容没变，也会误判为需要更新
  2. ETag / If-None-Match（检测资源的唯一标识）
    1. 响应头：ETag："abc123"
    2. 请求头：If-None-Match："abc123"
    3. ETag 是文件内容的哈希值或唯一标识。服务器比对是否一致，一致则返回 304。
    4. 优点：比时间更精确，内容没变就不会重新下载。
3 Cache-Control的常见取值
Cache-Control 指令可以单独或组合使用，以定义特定资源的缓存策略
1. no-store：禁止缓存。
2. no-cache： 需要重新验证缓存。客户端向服务端发送一个请求来确认缓存的有效性。
3. public：客户端和代理服务器都能缓存
4. private：只能被浏览器（客户端）缓存
5. immutable：指示响应不会随时间的推移而发生更改，适用于长期缓存的不变资源
53. 什么是同源策略
同源：协议（protocol）+ 域名（host）+ 端口（port） 三者都相同。只有同源的资源之间才能相互访问，防止恶意网站通过 <script> 或 AJAX 获取隐私信息。
同源策略：当一个源的文档和脚本，与另一个源的资源进行通信时，会对这个通信做出不同程度的限制。
会限制下面几类操作：
  1. Cookie / LocalStorage / IndexDB 的读取
  2. DOM 节点 的访问；
  3. AJAX 请求响应结果 的读取
  4. 脚本限制:不同源的脚本文件(如JavaScript)不能相互引用和执行

55. CORS 的简单请求和预检请求
1. 预检请求：当浏览器发现一个跨域请求可能有风险时，在真正发出请求之前，先发一个OPTIONS请求给服务器，如果服务器回答OK，浏览器才会继续发送真正的请求
2. 为什么会出现预检请求？
  1. 出于安全考虑，确认请求安全之后再放行正式请求
  2. 防止浏览器误操作或者资源泄露
3. 什么时候会触发预检请求——不是简单请求的都是预检请求
  简单请求要求：
  - 请求方法是GET、POST、HEAD之一
  - 头部字段满足CORS安全规范，详见 W3C（及使用默认头部字段，不改动和新增头部）
  - 如果有Content-Type，必须是下列值中的一个
    - text/plain  文本纯字符串提交
    - multipart/form-data 表单上传文件时使用的类型（文本+文件）
    - application/x-www-form-urlencoded 默认的表单提交格式
4. 优化
  1. 使用简单请求
  2. 缓存预检结果：Access-Control-Max-Age: 86400  # 单位：秒，表示1天内不再预检
  3. 同域部署 / Nginx 代理：用反向代理将接口转为同源，不触发 CORS。
  4. 使用 keep-alive / HTTP/2 复用连接：减少请求的额外 RTT 成本。
56. 线程和进程
进程
概念：一个正在运行的程序的实例（简单理解：一个程序运行需要专属的内存空间，这个内存空间可以被简单理解为进程），是系统资源分配的最小单位。
特点：
1. 独立的内存空间；
2. 进程之间相互隔离，相互独立（通信要双方同意）；
3. 启动开销大（独立资源）；
例子： 打开微信，微信会是一个进程；
线程
概念：进程中的执行单元，CPU调度和执行的最小单元。
主线程：一个进程至少有一个线程，进程开启后自动创建的一个线程。
如果程序需要同时执行多块代码，就会启动更多的线程来执行代码。
特点：
1. 共享进程的内存空间；
2. 线程之间切换开销小，适合并发处理；
3. 一个进程包含多个线程；
例子：微信的主界面是一个线程，接收消息是另一个线程，播放语音是另一个线程。
