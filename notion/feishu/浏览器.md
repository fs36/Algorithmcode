50. 重绘、回流（重排）？如何减少发生次数
1. 重绘：元素样式发生变化（元素的背景、字体颜色等），在不影响布局的情况下，浏览器重新绘制元素的过程
2. 回流（重排）：当页面的结构或几何属性（大小、位置）发生变化时，浏览器必须重新计算元素的布局，然后再重新渲染。
3. 回流的成本比重绘高，回流一定会触发重绘，而重绘不一定会触发回流
4. 如何减少回流？
  1. 合并多次的DOM操作：不频繁修改样式，用class一次性修改
  2. 批量修改 DOM：使用 DocumentFragment（文档片段） 批量插入 DOM 节点。在需要 批量插入/修改 DOM 的时候，创建文档片段const fragment = document.createDocumentFragment();，将这些元素添加到文档片段里面fragment.appendChild(li); ，再将这个文档片段一次性插入到DOM中DOM元素.appendChild(fragment);，减少回流次数。
  3. 使用 requestAnimationFrame ：浏览器会在下一帧渲染时统一执行，可以确保动画在浏览器的重绘周期内执行，从而避免不必要的回流
    1. requestAnimationFrame作用：告诉浏览器 在下一次重绘之前执行一个指定的回调函数，用来更新动画效果
  4. 使用visibility：hidden代替display：none：visibility：hidden不会触发回流，因为元素仍占据空间，只是不可见，display：none会将元素从渲染树移除，引起回流
  5. 让元素脱离文档流：修改元素前先 display: none，改完再显示 或者用 position: absolute/fixed 脱离文档流
  6. 减少强制同步布局：避免频繁读取 offsetHeight/getComputedStyle 后立刻修改 DOM，如果必须读取，尽量先缓存
5. 如何减少重绘？
  1. 用CSS动画代替JS动画：用transform和opacity来创建动画效果，不是改变元素的布局属性（宽、高等）。例如改变位置用transform：tanslate(x,y)，不用top/left，改变大小用scale(对元素进行缩放)，而不是width/height

52. 浏览器缓存（强缓存/协商缓存）

## 核心概念
**目的**：减少重复请求，提高页面加载速度，降低服务器压力。

**状态码区别**：
- 强缓存命中：返回 `200 OK (from disk cache)` 或 `200 OK (from memory cache)`
- 协商缓存命中：返回 `304 Not Modified`

**缓存流程图**：
```
浏览器发起请求
    ↓
检查强缓存（Cache-Control / Expires）
    ↓ 未过期
直接使用本地缓存 → 200 (from cache)
    ↓ 已过期
发起协商缓存请求（带 If-None-Match / If-Modified-Since）
    ↓
服务器对比资源是否变化
    ↓ 未变化              ↓ 已变化
返回 304                返回 200 + 新资源
使用本地缓存            更新本地缓存
```

---

## 1️⃣ 强缓存（Strong Cache）

### 特点
- 在缓存有效期内，浏览器**直接使用本地缓存**，不发送请求到服务器
- **效率最高**，适合静态资源（JS、CSS、图片、字体等）
- 开发者工具中显示 `200 (from disk cache)` 或 `200 (from memory cache)`

### 实现方式

#### ① Expires（HTTP/1.0，已过时）
```http
Expires: Wed, 30 Sep 2025 10:00:00 GMT
```
- **工作原理**：指定资源的过期时间（绝对时间）
- **缺点**：
  - 依赖客户端本地时间，如果本地时间不准确会导致缓存失效
  - 服务器和客户端时间不同步会出问题

#### ② Cache-Control（HTTP/1.1，推荐）
```http
Cache-Control: max-age=3600
```
- **工作原理**：指定资源的有效期（相对时间，单位：秒）
- **优点**：使用相对时间，不受本地时间影响
- **优先级**：Cache-Control 优先级高于 Expires

### 常见取值组合

| 指令 | 含义 | 适用场景 |
|------|------|----------|
| `max-age=31536000` | 缓存 1 年 | 带哈希的静态资源（如 `app.abc123.js`） |
| `no-cache` | 每次都需要验证 | 需要实时性的 HTML 文件 |
| `no-store` | 禁止缓存 | 敏感数据（用户信息、支付接口） |
| `public` | 可被任何中间节点缓存 | CDN 资源 |
| `private` | 只能被浏览器缓存 | 用户个人数据 |
| `immutable` | 资源永不改变 | 带版本号的静态资源 |
| `must-revalidate` | 过期后必须验证 | 重要的业务数据 |

### 实战示例

```javascript
// Node.js / Express 设置强缓存
app.get('/static/*', (req, res) => {
  // 静态资源缓存 1 年
  res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
  res.sendFile(filePath);
});

app.get('/index.html', (req, res) => {
  // HTML 文件不缓存，每次都验证
  res.setHeader('Cache-Control', 'no-cache');
  res.sendFile('index.html');
});
```

## 2️⃣ 协商缓存（Negotiation Cache）

### 特点
- 浏览器**先发请求询问服务器**，由服务器决定是否使用缓存
- 如果资源未变化，返回 `304 Not Modified`，浏览器使用本地缓存
- 如果资源已变化，返回 `200 OK` + 新资源内容
- **适合接口数据、频繁更新的资源**

### 实现方式

#### ① Last-Modified / If-Modified-Since（基于时间）

**工作流程**：
```http
# 首次请求
← 响应头：Last-Modified: Wed, 30 Sep 2025 10:00:00 GMT

# 再次请求
→ 请求头：If-Modified-Since: Wed, 30 Sep 2025 10:00:00 GMT
← 服务器对比时间
  - 未修改 → 304 Not Modified
  - 已修改 → 200 OK + 新资源 + 新的 Last-Modified
```

**缺点**：
1. **时间精度问题**：只能精确到秒级，1 秒内多次修改无法识别
2. **误判问题**：文件修改时间变了，但内容未变（如重新保存），也会重新下载
3. **时区问题**：不同服务器时区可能导致时间不一致

#### ② ETag / If-None-Match（基于内容，推荐）

**工作流程**：
```http
# 首次请求
← 响应头：ETag: "abc123def456"

# 再次请求
→ 请求头：If-None-Match: "abc123def456"
← 服务器对比 ETag
  - 一致 → 304 Not Modified
  - 不一致 → 200 OK + 新资源 + 新的 ETag
```

**ETag 生成方式**：
- 文件内容的哈希值（MD5/SHA1）
- 文件大小 + 最后修改时间的组合
- 文件的 inode + 大小 + 修改时间（Nginx 默认）

**优点**：
1. **精确度高**：基于内容而非时间，内容不变则 ETag 不变
2. **避免误判**：只要内容相同，即使修改时间变了也不会重新下载
3. **优先级更高**：同时存在时，ETag 优先于 Last-Modified

**缺点**：
- 服务器计算 ETag 有性能开销（尤其是大文件）
- 分布式系统中不同服务器生成的 ETag 可能不一致

### 实战示例

```javascript
// Node.js / Express 实现协商缓存
const crypto = require('crypto');
const fs = require('fs');

app.get('/api/data', (req, res) => {
  const data = getDataFromDB();
  const content = JSON.stringify(data);

  // 生成 ETag（内容的 MD5 哈希）
  const etag = crypto.createHash('md5').update(content).digest('hex');

  // 检查客户端的 If-None-Match
  const clientETag = req.headers['if-none-match'];

  if (clientETag === etag) {
    // 内容未变，返回 304
    res.status(304).end();
  } else {
    // 内容已变，返回新数据
    res.setHeader('ETag', etag);
    res.setHeader('Cache-Control', 'no-cache'); // 允许缓存但需验证
    res.json(data);
  }
});
```

## 3️⃣ Cache-Control 完整指令表

| 指令 | 含义 | 使用场景 |
|------|------|----------|
| `max-age=<秒>` | 资源有效期（相对时间） | 所有可缓存资源 |
| `s-maxage=<秒>` | CDN/代理服务器的缓存时间 | CDN 资源 |
| `no-cache` | 可以缓存，但每次使用前需验证 | HTML、API 接口 |
| `no-store` | 禁止任何缓存 | 敏感数据、实时数据 |
| `public` | 可被浏览器和代理服务器缓存 | 公共静态资源 |
| `private` | 只能被浏览器缓存，不能被代理缓存 | 用户个人数据 |
| `immutable` | 资源永不改变（即使刷新也不验证） | 带哈希的静态资源 |
| `must-revalidate` | 过期后必须向服务器验证 | 重要业务数据 |
| `proxy-revalidate` | 代理服务器过期后必须验证 | CDN 资源 |
| `stale-while-revalidate=<秒>` | 过期后先返回旧缓存，后台验证 | 提升用户体验 |
| `stale-if-error=<秒>` | 服务器错误时使用过期缓存 | 容错处理 |

### 组合使用示例

```http
# 静态资源（带哈希）：永久缓存
Cache-Control: public, max-age=31536000, immutable

# HTML 文件：每次验证
Cache-Control: no-cache

# API 接口：不缓存
Cache-Control: no-store, no-cache, must-revalidate

# CDN 资源：浏览器缓存 1 天，CDN 缓存 1 年
Cache-Control: public, max-age=86400, s-maxage=31536000

# 用户数据：仅浏览器缓存 5 分钟
Cache-Control: private, max-age=300, must-revalidate
```

## 4️⃣ 最佳实践

### 静态资源缓存策略

```
项目结构：
├── index.html          → Cache-Control: no-cache
├── assets/
│   ├── app.abc123.js   → Cache-Control: public, max-age=31536000, immutable
│   ├── style.def456.css → Cache-Control: public, max-age=31536000, immutable
│   └── logo.png        → Cache-Control: public, max-age=31536000
```

**核心思路**：
1. **HTML 不缓存**：`no-cache`，确保每次都能获取最新的资源引用
2. **带哈希的静态资源永久缓存**：`max-age=31536000, immutable`
3. **构建工具自动生成哈希**：Webpack/Vite 的 `contenthash`

### Nginx 配置示例

```nginx
server {
  # HTML 文件：不缓存
  location ~* \.html$ {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
  }

  # 带哈希的静态资源：永久缓存
  location ~* \.(js|css|png|jpg|jpeg|gif|svg|woff|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }

  # API 接口：协商缓存
  location /api/ {
    add_header Cache-Control "no-cache";
    # ETag 默认开启
  }
}
```

### 前端开发注意事项

```javascript
// ❌ 错误：强制不缓存会影响性能
fetch('/api/data', {
  cache: 'no-store' // 每次都重新请求
});

// ✅ 正确：让浏览器根据响应头决定
fetch('/api/data', {
  cache: 'default' // 遵循 HTTP 缓存策略
});

// ✅ 特殊场景：强制刷新
fetch('/api/data', {
  cache: 'reload' // 忽略缓存，重新请求
});
```

## 5️⃣ 面试高频问题

### Q1: 强缓存和协商缓存的区别？
| 对比项 | 强缓存 | 协商缓存 |
|--------|--------|----------|
| 是否发请求 | 不发请求 | 发请求 |
| 状态码 | 200 (from cache) | 304 Not Modified |
| 效率 | 最高 | 较高 |
| 适用场景 | 静态资源 | 动态数据、API |
| 响应头 | Cache-Control, Expires | ETag, Last-Modified |

### Q2: 用户强制刷新（Ctrl+F5）会怎样？
- **强缓存失效**：浏览器会在请求头加 `Cache-Control: no-cache`
- **协商缓存仍生效**：仍会发送 `If-None-Match` / `If-Modified-Since`
- **结果**：可能返回 304，也可能返回 200

### Q3: 如何彻底清除缓存？
1. **开发阶段**：打开 DevTools → Network → 勾选 "Disable cache"
2. **生产环境**：
   - 修改文件名（添加版本号或哈希）
   - 使用 `Cache-Control: no-store`
   - 清除浏览器缓存（用户操作）

### Q4: ETag 和 Last-Modified 能同时使用吗？
- **可以同时使用**，服务器会优先比对 ETag
- **ETag 优先级更高**：只要 ETag 匹配，就返回 304
- **兼容性考虑**：老旧浏览器可能不支持 ETag，保留 Last-Modified 作为降级方案
53. 什么是同源策略
同源：协议（protocol）+ 域名（host）+ 端口（port） 三者都相同。只有同源的资源之间才能相互访问，防止恶意网站通过 <script> 或 AJAX 获取隐私信息。
同源策略：当一个源的文档和脚本，与另一个源的资源进行通信时，会对这个通信做出不同程度的限制。
会限制下面几类操作：
  1. Cookie / LocalStorage / IndexDB 的读取
  2. DOM 节点 的访问；
  3. AJAX 请求响应结果 的读取
  4. 脚本限制:不同源的脚本文件(如JavaScript)不能相互引用和执行

2. CORS 跨域资源共享

55. CORS 的简单请求和预检请求
1. 预检请求：当浏览器发现一个跨域请求可能有风险时，在真正发出请求之前，先发一个OPTIONS请求给服务器，如果服务器回答OK，浏览器才会继续发送真正的请求
2. 为什么会出现预检请求？
  1. 出于安全考虑，确认请求安全之后再放行正式请求
  2. 防止浏览器误操作或者资源泄露
3. 什么时候会触发预检请求——不是简单请求的都是预检请求
  简单请求要求：
  - 请求方法是GET、POST、HEAD之一
  - 头部字段满足CORS安全规范，详见 W3C（及使用默认头部字段，不改动和新增头部）
  - 如果有Content-Type，必须是下列值中的一个
    - text/plain  文本纯字符串提交
    - multipart/form-data 表单上传文件时使用的类型（文本+文件）
    - application/x-www-form-urlencoded 默认的表单提交格式
4. 优化
  1. 使用简单请求
  2. 缓存预检结果：Access-Control-Max-Age: 86400  # 单位：秒，表示1天内不再预检
  3. 同域部署 / Nginx 代理：用反向代理将接口转为同源，不触发 CORS。
  4. 使用 keep-alive / HTTP/2 复用连接：减少请求的额外 RTT 成本。
56. 线程和进程
进程
概念：一个正在运行的程序的实例（简单理解：一个程序运行需要专属的内存空间，这个内存空间可以被简单理解为进程），是系统资源分配的最小单位。
特点：
1. 独立的内存空间；
2. 进程之间相互隔离，相互独立（通信要双方同意）；
3. 启动开销大（独立资源）；
例子： 打开微信，微信会是一个进程；
线程
概念：进程中的执行单元，CPU调度和执行的最小单元。
主线程：一个进程至少有一个线程，进程开启后自动创建的一个线程。
如果程序需要同时执行多块代码，就会启动更多的线程来执行代码。
特点：
1. 共享进程的内存空间；
2. 线程之间切换开销小，适合并发处理；
3. 一个进程包含多个线程；
例子：微信的主界面是一个线程，接收消息是另一个线程，播放语音是另一个线程。
