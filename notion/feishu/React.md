32. 什么是Hooks？Hooks 的优缺点？
1. 背景：Hooks 出现前，有几个问题：
  1. 函数组件功能受限：只能做渲染，复杂逻辑用类组件
  2. 类组件问题：生命周期里面混杂多种逻辑，代码难维护；this指向混乱；逻辑复用困难（组件嵌套过深）
  3. 状态逻辑分散：相关的业务逻辑（数据获取和处理）拆分到不同的生命周期里面，难以追踪
2. 核心目标：让函数组件拥有类组件的所有能力，同时解决逻辑复用和代码组织的问题。逻辑与UI分离，按功能组织，不是生命周期
3. 定义：以 use 开头的函数，让你在不编写类组件的情况下，使用 React 的状态、生命周期、副作用等核心特性。
4. 使用规则：保证每次渲染都按照同样顺序执行
  1. 只在顶层调用，保证每次的渲染顺序一致
  2. 只能在「函数组件」或「自定义 Hooks」中使用；保证Hooks的上下文有效
5. 常见的 Hooks 函数
  1. useState —— 状态管理
  2. useEffect —— 处理副作用
    1. [userId] 依赖数组，在userId变化时执行
    2. [] 空数组，只在挂载后执行
    3. 没有[]，每次渲染都执行
    4. 返回一个清理函数（return）在下一次effect执行前、组件卸载时执行
  3. useMemo ——缓存计算结果，当依赖值变化时才重新计算
  4. useCallBack —— 缓存函数，避免子组件不必要的重渲染，空依赖数组，函数永远不变
  5. useRef —— 获取DOM或存储渲染不变的值
  6. useContext —— 跨层级传递数据，实现组件间的状态共享
  7. useReducer ——复杂的状态逻辑管理
6. 自定义 Hooks 核心特点：
  - 以 use 开头，内部可以调用其他 Hooks；
  - 不直接渲染 UI，只封装「状态逻辑」；
  - 每次调用自定义 Hooks，都会创建独立的状态（不同组件使用不会共享状态）。
7. 核心优点：
  - 告别类组件的 this 问题，代码更简洁；
  - 自定义 Hooks 实现优雅的逻辑复用，无嵌套地狱；
  - 按业务逻辑聚合代码，维护性更高；
  - 函数组件从「纯渲染」升级为「全能组件」，TS 支持更好。
8. 主要缺点：
  - 有严格的使用规则，闭包陷阱、依赖数组易踩坑；
  - 心智负担更高（理解闭包、依赖、执行时机）；
  - 状态粒度细，多状态管理成本上升。
33. 什么是JSX？
1. 本质上是React.CreactElement的语法糖
2. 可以在html里面直接写js代码
3. 避免XSS注入攻击（会自动处理输入）
4. 定义：JSX 是 JavaScript 的语法扩展（语法糖），长得像 HTML，但本质是 JavaScript 函数调用，专门用来在 React 中描述 UI 结构。
5. 本质：所有的 JSX 会被 Babel 编译成 React.createElement函数调用，返回虚拟 DOM 对象
6. 核心规则：单根节点、驼峰属性名、{} 嵌入 JS 表达式、标签必须闭合；
7. 核心价值：简洁易读、类型安全、与 JS 无缝融合，跨平台兼容（JSX 不仅能编译成 DOM 节点，还能编译成原生组件，比如 React Native 中，JSX 会被编译成 iOS/Android 原生组件）
34. 函数组件和函数组件
本质区别：
1. 类组件：只实例化一次，保存了组件的state等状态，后续的每一次更新只需要调用render（）和对应的生命周期即可。
2. 函数组件：每一次更新都是一次函数执行，里面的变量会重新声明，为了使函数组件可以保存一些状态，执行一些副作用函数，出现了hooks，帮助处理react中组件的状态，处理额外的一些副作用
类继承优势如下：
- 可以控制父类 render，还可以添加一些其他的渲染内容；
- 可以共享父类方法，还可以添加额外的方法和属性。
但是也有值得注意的地方，就是 state 和生命周期会被继承后的组件修改。
35. 组件之间的通信方式
1. props 和 callback方式（父子通信）
  1. 父组件通过props传递数据，子通过执行props的callback来触发父组件信息的修改，实现通信
  2. 父组件 -> 通过自身 state 改变，重新渲染，传递 props -> 通知子组件
  3. 子组件 -> 通过调用父组件 props 方法 -> 通知父组件。
2. ref 实现组件通信
  1. 类组件：通过 ref 来获取子组件的实例，来实现组件通信
  2. 函数组件：forwardRef（允许函数组件接收 ref 参数） + useImperativeHandle（自定义暴露给父组件的实例值，）
// 父组件
function Parent() {
  const childRef = useRef(null);
  
  const handleClick = () => {
    // 调用子组件暴露的方法
    childRef.current?.doSomething();
    childRef.current?.updateValue('new value');
  };
  
  return (
    <div>
      <Child ref={childRef} />
      <button onClick={handleClick}>调用子组件方法</button>
    </div>
  );
}

// 子组件
const Child = forwardRef((props, ref) => {
  const [value, setValue] = useState('');
  
  // 向父组件暴露特定的方法
  useImperativeHandle(ref, () => ({
    doSomething: () => {
      console.log('子组件执行操作');
      setValue('updated');
    },
    updateValue: (newValue) => {
      setValue(newValue);
    },
    getValue: () => value
  }));
  
  return <div>{value}</div>;
});
3. 多层嵌套：context 跨层级组件通信，在多个组件之间共享全局状态
4. 复杂应用：全局状态管理（Redux、Zustand），多个组件之间共享状态的场景
5. 跨组件通信：EventEmitter ，通过自定义事件实现跨组件通信。ComponentA 通过 eventEmitter.emit 触发事件，ComponentB 通过 eventEmitter.on 监听事件
6. 通过 URL 参数（路由传递），在页面之间传递参数（如 id、筛选条件等）时使用
7. 兄弟通信：提升状态到父组件
36. 虚拟 DOM （Vue2 React）的意义
- 定义：一个轻量级的JS对象表示真实的DOM的结构和属性
- 原理：用 JS 模拟真实的DOM，通过 diff 算法找出最小更新，再批量操作
- 优点：跨平台，避免直接操作 DOM 的低效
- 缺点：运行时计算 diff 有开销
- 减少实际的 DOM 操作：通过比较新旧虚拟 DOM 树的差异， React 可以确定需要更新的部分，并生成最小化的DOM 操作序列。这样可以减少实际的 DOM 操作次数，提高性能。
- 批量更新： React 会将所有需要更新的 DOM 操作批量执行，从而避免了频繁的 DOM 操作，提高了性能。
- 跨平台兼容性：虚拟 DOM 是一个轻量级的 JavaScript 对象，可以在不同的平台上运行，例如浏览器、移动设备和服务器。这使得 React 可以在多个环境中使用相同的代码和逻辑。
- 更好的开发体验：虚拟 DOM 使得开发者可以使用类似于 HTML 的标记语言来描述 UI ，而不需要直接操作 DOM 。这简化了开发过程，并提供了更好的开发体验。
- 注意：Vue3 编译时优化：
  - 原理：在编译阶段把静态资源提升，动态节点标记，运行时直接更新动态部分。
  - 优点：运行时无虚拟 DOM 开销，性能更优
  - 缺点：框架自身复杂，编译产物对开发者不透明

38. Fiber
1. 背景：16 之前 diff 算法，一旦开始就会递归遍历整个DOM树，直到完成更新。如果组件树很深或者更新频繁，主线程会一直被占用导致页面卡顿。
2. Fiber 的核心目的： 把「不可中断的同步更新」改成「可中断、可恢复、可优先级排序的异步更新」，解决主线程阻塞的问题。
3. 定义：（两个维度理解）
  1. 数据结构：一个「链表节点对象」，用来替代原来的虚拟 DOM 节点，存储了组件的很多信息
    1. 类型、props、state等信息
    2. 指向其他 fiber 的指针，return（父节点）、child（第一个子节点）、sibling（兄弟节点），构成一个链表结构的树
    3. 用于双缓存的引用：alternate 指向上一次渲染的 fiber
    4. 有关调度的元数据
// 简化的 Fiber 节点结构
const FiberNode = {
  // 1. 组件相关信息
  type: 'div', // 组件类型（比如 div、FunctionComponent、ClassComponent）
  props: {},   // 组件接收的 props
  stateNode: null, // 对应的真实 DOM 节点（比如 div 元素）
  
  // 2. 链表关系（替代原来的树结构，方便中断/恢复）
  parent: null,   // 父 Fiber 节点
  child: null,    // 第一个子 Fiber 节点
  sibling: null,  // 下一个兄弟 Fiber 节点
  return: null,   // 指向父节点（和 parent 作用一致，是 Fiber 规范的命名）
  
  // 3. 任务调度相关
  tag: 0,         // 节点类型标记（比如函数组件、类组件、宿主组件）
  effectTag: 0,   // 标记该节点需要执行的操作（比如插入、更新、删除）
  expirationTime: 0, // 任务优先级（过期时间，数字越小优先级越高）
  
  // 4. 状态相关
  memoizedState: null, // 组件的当前状态（比如 useState 的值）
  updateQueue: null,   // 组件的更新队列（比如 setState 的待执行任务）
};
  2. 执行单元维度：Fiber 是一个「可中断的工作单元」，每个 Fiber 节点对应一个组件的「更新任务」（比如 Diff、DOM 更新），这个任务可以执行一点就暂停，等主线程空闲了再继续。
4. 工作原理：Fiber 把 React 的更新流程拆成了两个独立阶段，核心是 阶段 1 可中断，阶段 2 不可中断
  阶段 1：render（协调阶段 / 渲染阶段）父->子
  1. 核心：遍历 Fiber 树，执行 Diff 算法，标记每个 Fiber 节点的「更新操作」（比如 effectTag 标记为 UPDATE/DELETE/INSERT）；
  2. 关键特性：可中断、可恢复、可优先级排序——React 会利用浏览器的「空闲时间」执行这个阶段的任务，一旦主线程有更高优先级的任务（比如用户输入、动画），就暂停当前 Fiber 任务，等主线程空闲了再从暂停的位置继续。
  3. 执行方式：通过「链表遍历」（深度优先：当所有的子节点处理完，再处理兄弟节点或者父节点）替代原来的「递归遍历」，因为递归一旦开始无法中断，而链表可以记录当前执行到哪个节点，方便暂停 / 恢复（通过指针实现）。节点的处理使用 diff 算法
  4. 遍历过程使用双缓存机制（current树（当前UI），WIP（WorkInProcess）树（构建的新Fiber树））
    优点：
    1. 在构建新树的同时，旧树仍然响应用户事件
    2. 同时WIP可以复用current树的节点
  5. 单节点，通过比较type和key来决定复用还是重建
  6. 多节点（如数组形式的children），遍历新旧列表，用key匹配节点，并识别移动、插入、删除。
  7. 通过标记effectTag，diff的结果被记录，留到commit阶段执行
  阶段 2：Commit（提交阶段）子->父
  - 核心工作：根据阶段 1 标记的 effectTag，执行真实 DOM 的更新（插入、修改、删除），同时执行生命周期钩子（比如 componentDidMount、useEffect）；
  - 关键特性：不可中断—— 这个阶段操作真实 DOM，必须一次性执行完，否则会导致 DOM 状态不一致；
  - 执行速度：因为阶段 1 已经完成了所有计算和标记，这个阶段的操作非常快，几乎不会阻塞主线程。

40. 什么是副作用？如何处理副作用
1. 编程定义：函数执行时，除了返回值之外，对外部环境产生的任何可观察的改变，例如（改变量、发请求、操作 DOM、调定时器等）
2. React：在 React 中，组件的核心职责是「根据 state/props 渲染 UI」（这是纯计算，无副作用），但实际开发中，我们需要在组件中做「渲染之外的事」—— 这些事就是 React 中的「副作用」。
暂时无法在飞书文档外展示此内容
3. 处理副作用的原因：
  1. 组件的渲染是可中断、可重复执行的，在组件里面写副作用，会导致
    1. 副作用被重复执行（比如每次渲染都发请求、都加定时器）；
    2. 副作用执行时机不可控（比如 DOM 还没渲染完成就操作 DOM）；
    3. 内存泄漏（比如组件卸载了，定时器 / 监听还在运行）。
4. 使用useEffect处理副作用
// 副作用：发起网络请求获取用户列表
useEffect(() => {
// 1. 执行副作用（组件挂载后发请求）
const fetchUsers = async () => {
const res = await fetch('/api/users');
const data = await res.json();
setUsers(data);
};
fetchUsers();
// 2. 清理副作用（组件卸载时执行）
return () => {
// 比如取消请求、清理定时器等
console.log('组件卸载，清理副作用');
};
}, []); // 依赖数组：空数组 → 只在组件挂载/卸载时执行
5. 副作用的「执行时机」和「清理」
  - 执行时机：默认在「组件渲染完成后」执行（保证 DOM 已挂载，可安全操作）；
  - 依赖数组：控制副作用的重新执行时机：
    - []：只在「组件挂载」时执行一次，「卸载」时执行清理；
    - [count]：只在 count 变化时执行；
    - 无依赖数组：每次组件渲染都执行；
  - 清理函数：useEffect 返回的函数会在「副作用重新执行前」或「组件卸载时」执行，用来清理旧的副作用（比如清除定时器、取消监听）。

41. 自定义 hooks 的思路
核心思路：发现重复--提取逻辑--配置化--稳定输出
思路路径：
1. 提取重复代码
2. 按业务划分
3. 配置化驱动（传入配置实现通用性）
4. 依赖注入（不确定的实现通过参数传入）
设计原则
- 单一职责：一个 hook 只做一件事

- 类型安全：完整的 TS 类型定义
- 错误处理：内部处理常见错误 —— loading状态、try-catch

1. 识别重复（分析哪些组件有相同的逻辑可以复用）
2. 确定输入-参数（什么会变化？API？配置？回调？）
3. 确定输出-返回值（返回什么状态和方法）
4. 类型定义
5. 内部实现（单一职责）
6. 暴露稳定引用
